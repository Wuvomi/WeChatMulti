# FDA(全盘访问)检测 — 现状、疑点、验证方法

## 问题
注入进微信(沙盒 App)的探针,要准确判断"微信是否被授予了完全磁盘访问(FDA)",一直没闭环验证成功。

## 已试过的方法(都未在"微信实际带引擎运行 + 确有 FDA"下验证过)
- **0.9.1**:读系统 `/Library/Application Support/com.apple.TCC/TCC.db`,能读到=有 FDA。
- **0.9.2**:改读 `~/Library/Safari`(用户级、受 FDA 保护、相对无害),能读到=有 FDA。

## 核心疑点(待证实)
**App Sandbox 应用即便被授予 FDA,其进程能否真的读到受保护文件?** 沙盒(seatbelt profile,启动即定)与 FDA(TCC 授权)是两层;微信的沙盒 profile 可能本就不允许读 ~/Library/Safari、TCC.db,FDA 这层 TCC 例外未必能覆盖沙盒层限制。若如此,"读保护文件判 FDA"对沙盒微信根本不成立,需换思路或放弃注入式检测。

## 为什么夜间没能自动验证
1. 用户截图时微信"已打开 0 个"——**微信没在跑**,0.9.2 探针没机会执行,perms.json 是旧数据(红是旧的,不代表 0.9.2 真失败)。
2. 要验证"有 FDA → 探针返回 true",需要一个**已授予 FDA**的 App 实例;无法程序化给 /tmp 测试 App 授 FDA(TCC 授权受保护、需人工在系统设置里加),所以自动闭环受阻。
3. 深入逆向"沙盒 vs FDA"的 subagent 任务被 Opus 安全策略拦截(TCC/沙盒探测框架触发网络安全 flag),此路 subagent 走不通。

## 用户醒来的验证步骤(一次就能定论)
1. 确认微信装的是 **0.9.2** 自研引擎(GUI"双开插件版本"显 0.9.2;否则先点"更新双开引擎")。
2. **打开微信并让它真正运行**(GUI"已打开的微信"≥1)。
3. 确认系统设置 > 完全磁盘访问里**微信是开的**。
4. 看容器内 perms.json 的 `fda` 字段:
   `~/Library/Containers/com.tencent.xinWeChat/Data/Library/Application Support/WeChatMulti/perms.json`
   - 若 `fda:true` → 0.9.2 的 Safari 探针**成功**,检测可用,GUI 应显绿/隐藏。
   - 若仍 `fda:false` → 沙盒确实挡住了保护文件读取,**注入式 FDA 检测此路不通**,需走下面的备选。

## 若注入式检测确认不可行 — 备选方案(待用户定夺)
- **A. 放弃红/绿检测**,FDA 行只保留中性引导(去设置 + 橙色提示),不再声称"已检测";用户授权一次、弹窗消失即可。最诚实简单。
- **B. 换非保护性的判定信号**(如检测 cdhash 变化后是否需重新授权的某个间接迹象)——需进一步研究,且要避开敏感框架。
- **C. 逆向微信本身**怎么判断自己没权限(它弹"想访问其他App数据"),抄它的判据——较深,且框架敏感。

> 当前 GUI 行为:perms.json 不新鲜(微信没跑)时,FDA 显示是旧值,可能误红。已知局限,待上面验证后再定最终方案。
