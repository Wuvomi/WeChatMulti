# 自研引擎破坏微信视频播放 — bug 记录与修复方向

## 现象
装上自研引擎后,微信里点开视频只剩预览图、点播放按钮无反应。**卸载引擎(还原干净业务体 + 还原 loader 字节)后视频恢复正常。** X1a0He 装着时视频正常。

## 已知差异(自研引擎 vs X1a0He)
自研引擎比 X1a0He 多做的、且最可能是根因的:
1. **byte-patch 了 loader**(门①,mach 单例分支)——X1a0He 从不碰 loader。怀疑改 loader 二进制破坏了微信的媒体/视频子系统(loader 重签后某媒体 helper 加载失败,或 loader 初始化链被动到)。
2. 注入 dylib(两者都做,但实现不同)。

## 修复方向(待交互式验证)
**关键假设:门① loader patch 很可能不必要。** 据 `re/self-engine-v2.md`,真正放行第二实例的是**门②(运行时 NOP `tbz w20@0x2117e0`)**;门①(loader 静态 patch)当时是和门② 一起上的,**没单独验证过"去掉门① loader patch、只靠门② + 注入"能否多开**。

→ **要做的实验(/tmp):** 安装时跳过门① loader patch(只注入引擎 dylib + 重签;门② 仍由 constructor 运行时在第二实例 NOP),测 `open -n` 同路径 2 实例能否并存稳定。
- **若能** → 门① 不必要,去掉它 → 既消除对 loader 的二进制修改(很可能就是视频 bug 根因),又让引擎更净(不碰 loader)。
- **若不能** → 门① 必需,需让 loader 改动不破坏媒体(更保守重签、保留 loader 原 entitlements/flags)。

辅助:对比"自研装好 / X1a0He装好 / 干净"三者的 `codesign -dvvv --entitlements`、`codesign --verify --deep --strict`,找签名异常作佐证。

## 为什么没自动修
自研引擎的二进制修改(byte-patch / 注入 / 为注入重签)这类任务,**交给 subagent 会被 Opus 安全策略拦截**(已两次:FDA 探测、本视频修复)。需**交互式**完成(用户在场逐步做),不能用后台 subagent。

## 当前缓解(已实装)
- **自研引擎设为默认关闭的实验选项**(GUI「自研引擎（实验）」复选框,默认不勾)→ 全新安装默认装 **X1a0He(不破坏视频)**。
- 用户日常使用安全;视频 bug 只影响主动勾选实验引擎者。
- 复测方法:装自研引擎 0.9.x → 微信里点视频播放;改成"不patch loader"的版本后再测,看视频是否恢复。
